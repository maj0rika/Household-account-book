# N분의 1 정산 기능 PM 플래닝

## 최우선 불변조건

이 기능의 핵심은 **총무 역할을 했을 때도 내 가계부가 더럽혀지지 않는 것**이다.

아래 규칙은 협상 대상이 아니라 구현 전반에서 반드시 지켜야 하는 불변조건이다.

- 총무가 실제로 `총액`을 결제했더라도, 가계부 본체의 거래 금액은 항상 `내 부담금`만 기록한다.
- 총액, 참여자 몫, 미수금, 회수 상태는 정산 레이어에서만 관리한다.
- 친구가 보낸 정산금은 기본적으로 `수입 거래`로 저장하지 않는다.
- 월 요약, 카테고리 통계, 예산, 거래 리스트는 절대로 `총액`이나 `정산 회수액`을 합산하지 않는다.

## 기능 요약

총무가 먼저 전체 금액을 결제하더라도, **가계부 본체에는 내 부담금만 기록**하고, 나머지 금액은 **정산 추적 레이어**에서 관리하는 `n/1` 기능을 설계한다.

## 문제 정의

현재 구조는 `transactions.amount` 하나가 거래 목록, 월 요약, 카테고리 통계, 계좌 잔액 반영까지 모두 담당한다.

이 상태에서 총무 상황의 `총액`을 그대로 거래 금액으로 저장하면 아래 문제가 생긴다.

- 내 실제 소비보다 지출 통계가 과대 집계된다.
- 카테고리별 지출, 월 예산, 대시보드 수치가 왜곡된다.
- 자연어 입력이 편하더라도 결과가 사용자의 정신 모델과 어긋난다.

반대로 `내 몫`만 저장하면 가계부는 깔끔해지지만, 실제 결제 총액과 친구들에게 받을 돈을 추적할 수 없다.

따라서 **가계부 금액**과 **정산 메타데이터**를 분리해야 한다.

## PM 결론

### 1. 가계부 본체는 `내 부담금` 기준

- `transactions.amount`는 계속해서 `내가 최종 부담하는 금액`으로 유지한다.
- 월 요약, 카테고리 통계, 예산 비교, 거래 리스트는 이 값만 사용한다.
- `총액`은 정산용 보조 정보이며, 거래 본체와 분리 저장한다.

### 2. 정산은 별도 도메인으로 관리

- 총액, 참여 인원, 참여자별 몫, 총무 여부, 미수금/완료 상태는 `settlement` 계층에서 관리한다.
- 다른 사람이 내게 보내는 돈은 기본적으로 `수입` 거래가 아니라 `정산 회수` 이벤트다.
- 즉, 정산 회수는 가계부 수입 통계를 오염시키지 않아야 한다.

### 3. 자연어와 이미지 파싱이 핵심 진입점

- 텍스트 입력: `어제 4명이서 회 12만원, 내가 카드로 냈고 내 몫은 3만원`
- 이미지 입력: 카카오톡/토스/기타 서비스의 `n/1`, 정산 요청, 송금 현황 스크린샷
- 둘 다 파싱 후 바로 저장하지 않고, 반드시 수정 가능한 중간 확인 단계가 필요하다.

### 4. 파싱 오류는 편집으로 복구 가능해야 함

- AI가 `총액`, `내 몫`, `인원 수`, `총무 여부`, `참여자 이름`, `완료 상태`를 잘못 읽어도 사용자가 저장 전에 수정할 수 있어야 한다.
- 저장 후에도 거래 상세 또는 정산 상세에서 수정 가능해야 한다.

## 사용자 시나리오

### 시나리오 A. 총무가 자연어로 입력

입력:

`어제 3명이서 마라탕 45000원, 내가 카드로 계산했고 내 몫은 15000원`

기대 결과:

- 거래 1건 생성: 지출 `15000원`
- 정산 1건 생성: 총액 `45000원`, 인원 `3명`, 내 몫 `15000원`, 받을 돈 `30000원`
- 거래 화면은 `15000원`만 보이고, 정산 상세에서 총액과 미수금이 보인다.

### 시나리오 B. 총무가 스크린샷 업로드

입력:

- 카카오톡 또는 토스의 정산 요청/완료 화면 스크린샷

기대 결과:

- 서비스명, 총액, 참여자 수, 참여자 이름, 완료/미완료 상태를 최대한 자동 인식
- 파싱 결과 시트에서 사용자 수정 후 저장

### 시나리오 C. 참여자 입장 입력

입력:

`어제 곱창 28000원 정산해서 내 몫 14000원 보냈어`

기대 결과:

- 내 지출은 `14000원`만 기록
- 정산 대시보드에는 참여 기록만 남기거나, 단순 거래로 저장할지 선택 가능

### 시나리오 D. 정산금 회수

행동:

- 친구가 돈을 보내서 정산 멤버를 `완료` 처리

기대 결과:

- 정산 미수금이 감소
- 필요 시 수령 계좌 잔액은 증가
- 가계부 `수입` 통계에는 기본적으로 포함되지 않음

## 핵심 요구사항

1. 자연어 입력으로 `일반 거래`와 `정산 포함 거래`를 함께 처리할 수 있어야 한다.
2. 이미지 업로드로 정산 스크린샷을 파싱할 수 있어야 한다.
3. 저장되는 가계부 거래 금액은 항상 `내 부담금` 기준이어야 한다.
4. `총액`, `인원`, `참여자별 몫`, `총무 여부`는 별도 정산 데이터로 저장되어야 한다.
5. 파싱 결과는 저장 전 수정 가능해야 한다.
6. 저장 후에도 정산 상태(받아야 함/완료)를 수정할 수 있어야 한다.
7. 정산 회수 금액은 기본적으로 수입 통계에 포함되지 않아야 한다.
8. `총액 기록`은 옵션이어야 하며, 사용자가 내 몫만 빠르게 기록하는 흐름도 지원해야 한다.

## 수용 기준

- [ ] 총무가 `총액 90000원, 3명, 내 몫 30000원` 상황을 저장해도 월 지출 합계에는 `30000원`만 반영된다.
- [ ] 거래 목록/대시보드/카테고리 통계는 정산 총액이 아니라 `내 부담금`만 사용한다.
- [ ] 정산 상세 화면에서 총액, 참여자, 미수금, 완료 상태를 확인할 수 있다.
- [ ] 카카오톡/토스 등 정산 스크린샷 업로드 후 초안이 생성되고, 저장 전에 수정할 수 있다.
- [ ] AI 파싱이 틀려도 사용자가 `총액`, `내 몫`, `인원`, `참여자 이름`을 직접 수정할 수 있다.
- [ ] 친구가 입금한 뒤 멤버를 완료 처리하면 정산 현황이 즉시 갱신된다.
- [ ] 정산 회수는 기본 수입 통계에 포함되지 않는다.

## 권장 데이터 모델

### 원칙

- `transactions.amount` = 개인 가계부 기준 금액 (`내 부담금`)
- 정산 총액/참여자/상태 = 별도 `settlements` 도메인

### 권장 스키마 초안

- `transactions`
  - 기존 `amount` 유지
  - 정산 연결용 `settlementId` 또는 settlement 쪽의 `transactionId` 필요
  - 계좌 정합성을 위해 `accountImpactAmount` 분리 여부 검토 필요
- `settlements`
  - `id`
  - `userId`
  - `transactionId`
  - `title`
  - `totalAmount`
  - `myShareAmount`
  - `participantCount`
  - `isOrganizer`
  - `status` (`pending` | `partial` | `completed`)
  - `sourceType` (`text` | `image` | `manual`)
  - `sourceService` (`kakao`, `toss`, `unknown` 등)
- `settlementMembers`
  - `id`
  - `settlementId`
  - `name`
  - `shareAmount`
  - `status` (`pending` | `paid`)
  - `paidAmount`
  - `paidAt`

### 계좌 정합성 리스크

현재 계좌 잔액은 거래 생성 시 `transactions.amount` 기준으로 즉시 반영된다.

하지만 총무는 실제로 `총액`이 빠져나가고, 가계부 지출만 `내 몫`으로 남겨야 한다. 따라서 아래 둘 중 하나가 필요하다.

- `transactions.amount`와 별도로 `accountImpactAmount`를 둔다.
- 또는 정산 저장 시 실제 결제/회수 금액을 계좌에 반영하는 별도 정산 액션을 둔다.

PM 권장안은 **통계/가계부 금액과 계좌 영향 금액을 분리**하는 것이다. 그렇지 않으면 계좌 기능을 함께 쓰는 사용자의 잔액이 틀어진다.

## 파싱 설계 방향

### 텍스트 파싱

`ParsedTransaction`에 아래 메타데이터를 추가하는 방향이 적절하다.

- `isSettlement?: boolean`
- `settlementRole?: "organizer" | "participant"`
- `settlementTotalAmount?: number | null`
- `myShareAmount?: number | null`
- `participantCount?: number | null`
- `members?: { name: string; shareAmount?: number; status?: "pending" | "paid" }[]`

### 이미지 파싱

현재 이미지 업로드 파이프라인은 이미 존재하므로, 정산 화면 인식 규칙을 추가해 확장하는 방식이 유리하다.

- 카카오톡 `정산하기`, `더치페이`, 송금 완료/대기 화면
- 토스 `n/1`, 정산 요청, 송금 내역 화면
- 기타 서비스는 `unknown`으로 받아도 저장 직전 편집 가능해야 한다.

### 후속 파싱 항목

이번 Phase 핵심은 `정산 생성`과 `정산 상태 추적`이다.

후속으로 아래 자동화 파싱을 추가할 수 있다.

- 정산 완료 메시지 파싱
- 입금 알림 메시지와 정산 건 자동 매칭
- 스크린샷 재업로드로 상태 동기화

## 추천 구현 원칙

1. `가계부`와 `정산`을 한 테이블에서 억지로 해결하지 않는다
2. 사용자에게 보이는 핵심 금액은 항상 `내가 실제 부담한 금액`이다
3. 총액과 정산 상태는 참고/추적용 정보로 분리한다
4. AI 파싱은 초안 생성기일 뿐이며, 저장 전 수정 가능성이 반드시 열려 있어야 한다
5. 이미지 파싱은 코어 기능이지만, 정확도보다 `수정 가능한 초안`이 우선이다
