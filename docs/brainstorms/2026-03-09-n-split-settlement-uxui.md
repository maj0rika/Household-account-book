# N분의 1 정산 기능 UX/UI 설계

## 최우선 UX 원칙

이 기능은 `총무를 했다는 이유만으로 가계부 화면이 총액 기준으로 오염되지 않게` 설계해야 한다.

따라서 사용자에게 보이는 기본 금액 체계는 아래처럼 고정한다.

- 거래 리스트, 월 요약, 차트, 예산 화면의 기본 금액은 항상 `내 부담금`
- `총액`, `받을 돈`, `정산 진행 상태`는 정산 전용 정보로 분리
- 정산 회수는 기본적으로 수입으로 보이지 않음

## 설계 목표

`n/1 정산`은 거래 입력의 확장 기능이지, 기존 가계부 입력 흐름을 방해하는 별도 앱이 아니다.

따라서 UX 목표는 아래 4가지다.

1. 사용자는 평소처럼 자연어 또는 이미지를 입력한다
2. AI가 `정산 포함 거래`를 감지하면 저장 전 수정 가능한 초안을 보여준다
3. 가계부에는 `내 부담금`만 남고, 정산은 별도 화면에서 추적된다
4. 진행 중 정산은 거래 화면 안에서 자연스럽게 다시 발견된다

## UX 핵심 결정

### 1. 정산은 글로벌 탭이 아니라 거래의 하위 흐름

모바일 하단 탭은 이미 `홈 / 분석 / 입력 / 자산 / 설정` 5개로 꽉 차 있다.

정산을 별도 하단 탭으로 승격하면 다음 문제가 생긴다.

- 1차 입력 흐름보다 보조 흐름이 과도하게 전면에 나온다
- 탭 밀도가 높아져 한 손 조작성이 떨어진다
- 정산을 쓰지 않는 사용자에게 상시 노이즈가 된다

따라서 초기 UX는 아래처럼 설계한다.

- 모바일: `/transactions` 상단의 `정산 요약 카드`와 거래 아이템의 `정산 배지`에서 진입
- 데스크톱: 동일 진입 구조를 유지하고, 필요 시 후속 단계에서 Sidebar 승격 검토
- 별도 라우트는 제공: `/settlements`

### 2. 저장 전 수정은 `거래 정보`와 `정산 정보`를 분리해 보여준다

현재 [ParseResultSheet.tsx](/Users/leeth/Documents/git/Household%20account%20book/src/components/transaction/ParseResultSheet.tsx)는 거래 편집에 최적화되어 있다.

정산까지 동일 밀도로 한 화면에 모두 노출하면 과밀해진다. 그래서 항목당 2단 구조를 사용한다.

- 1층: 거래 요약과 `내 부담금`
- 2층: 필요할 때만 펼치는 `정산 정보`

### 3. 총무와 참여자 UX를 다르게 설계한다

총무는 `받을 돈 추적`이 핵심이고, 참여자는 대부분 `내 몫 기록`이 핵심이다.

따라서 기본값을 다르게 둔다.

- 총무(`organizer`): `정산 추적` 기본 ON
- 참여자(`participant`): `거래만 저장` 기본 ON, `정산도 기록`은 선택 옵션

### 4. 이미지 파싱은 정확도보다 복구 가능성이 우선

카카오톡/토스 스크린샷은 레이아웃이 자주 바뀌고, OCR/비전 인식 오류가 생길 수 있다.

따라서 UX 목표는 `완벽 자동 저장`이 아니라 아래다.

- 서비스 인식
- 총액/인원/이름/상태 초안 생성
- 사용자가 빠르게 수정 후 저장

## 사용자 시나리오

### 시나리오 A. 총무가 자연어로 입력

1. 사용자가 `3명이서 마라탕 45000원, 내가 냈고 내 몫은 15000원`을 입력한다
2. 입력바는 기존처럼 파싱 상태 메시지를 보여준다
3. 결과 시트에서 거래 1건이 보이고, `정산 포함` 배지가 함께 노출된다
4. 사용자는 `내 지출 15000원`, `총액 45000원`, `인원 3명`, `받을 돈 30000원`을 확인/수정한다
5. 저장 시 거래 1건과 정산 1건이 함께 생성된다
6. 거래 화면 상단 정산 요약 카드에 `받을 돈 30000원`이 나타난다

### 시나리오 B. 총무가 정산 스크린샷을 업로드

1. 사용자가 카카오톡 또는 토스 정산 화면을 업로드한다
2. 결과 시트 상단에 `카카오 정산`, `토스 n/1`, `알 수 없는 서비스` 같은 출처 배지가 노출된다
3. 총액, 참여자 수, 멤버 이름, 완료/미완료 상태가 초안으로 채워진다
4. 사용자는 잘못 인식된 이름/금액만 고친 뒤 저장한다

### 시나리오 C. 참여자가 내 몫만 기록

1. 사용자가 `곱창 정산해서 14000원 보냈어`라고 입력한다
2. 결과 시트는 기본적으로 `거래만 저장` 상태를 선택한다
3. 사용자는 그대로 저장하거나, 필요하면 `정산도 기록` 스위치를 켜서 참여 기록을 남긴다

### 시나리오 D. 진행중 정산을 다시 관리

1. 사용자가 거래 화면에 진입한다
2. 월 요약 아래 `정산 요약 카드`에서 `받을 돈`, `보낼 돈`, `미완료 n건`을 본다
3. 카드를 누르면 `/settlements`로 이동한다
4. 카드 리스트에서 특정 멤버를 `완료 처리`한다
5. 해당 카드만 즉시 갱신되고, 상단 합계도 반영된다

## 세부 UX 규칙

### 1. 파싱 결과 시트

- 정산이 없는 일반 거래는 현재 UI를 유지한다
- 정산이 감지된 거래만 `정산 포함` outline 배지를 추가한다
- `내 부담금`은 기존 금액 영역에 유지하고, `총액`은 정산 섹션 안에서만 보여준다
- 총무일 때는 `받을 돈` 값을 거래 요약에서도 함께 보여준다
- 참여자일 때는 기본값을 `거래만 저장`으로 두고, 정산 추적 스위치를 세컨더리 액션으로 배치한다

### 2. 직접 입력 다이얼로그

- 기본 폼은 그대로 유지한다
- 하단에 `정산 포함` 스위치를 추가한다
- 스위치 ON 시에만 `총액`, `내 몫`, `인원`, `내가 결제했는지`가 나타난다
- 상세 멤버 입력은 이 다이얼로그에서 과도하므로 저장 후 상세 화면에서 보완한다

### 3. 거래 리스트

- 리스트는 가독성이 우선이므로 한 줄에 너무 많은 정산 정보를 넣지 않는다
- 보조 정보 노출 우선순위
  1. `정산` 배지
  2. 진행중인 총무 건이면 `미수 n원`
  3. 완료면 `정산 완료`

### 4. 정산 대시보드

- 기본 세그먼트: `전체 / 받을 돈 / 보낸 돈 / 완료`
- 카드 상단에는 상태와 날짜를, 카드 본문에는 제목/총액/내 몫/남은 금액을 둔다
- 멤버가 많아도 카드 자체는 2명까지만 노출하고 나머지는 `+3` 식으로 접는다
- `완료 처리`는 카드 안에서 바로 가능해야 한다
