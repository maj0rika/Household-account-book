<summary><h2>2. 기술 스택 — 왜 이걸 골랐는가</h2></summary>

| 기술 | 선택 이유 | 고려한 대안 | 트레이드오프 |
|------|-----------|------------|-------------|
| **Next.js 15** (App Router) | RSC(React Server Components)로 DB 직접 접근 → 별도 API 레이어 불필요, 서버/클라이언트 경계 명확. RSC란 서버에서만 실행되는 React 컴포넌트로, JS 번들에 포함되지 않아 클라이언트 성능에 영향 0 | Remix, SvelteKit | App Router 성숙도 이슈 존재하나, RSC의 생산성이 압도적 |
| **React 19** | Server Actions로 form mutation 간소화, `use()` 훅으로 비동기 처리 개선 | React 18 | 최신 버전이라 일부 라이브러리 호환성 주의 필요 |
| **Supabase** (PostgreSQL) | RLS(Row Level Security) 내장, 무료 티어 충분, Drizzle ORM과 궁합 좋음 | PlanetScale(MySQL), Firebase(NoSQL) | **금융 데이터는 관계형 DB가 필수**. Firebase/NoSQL의 장점은 실시간 동기화, 스키마리스로 빠른 프로토타이핑, 수평 확장 용이인데 — 가계부는 거래-카테고리-계좌 간 참조 무결성(FK), 트랜잭션(ACID), 집계 쿼리(SUM/GROUP BY)가 핵심이라 NoSQL이 맞지 않음 |
| **Drizzle ORM** | **TypeScript-first**: 스키마 정의가 곧 타입이라 별도 `generate` 불필요, SQL에 가까운 직관적 API | Prisma, Kysely | Prisma 장점: ORM 생태계 1위, 마이그레이션 자동화(`prisma migrate`), 직관적 스키마 언어, GUI 관리 도구(Prisma Studio). 그러나 `.prisma` 스키마와 TS 타입이 분리되어 `generate` 단계가 필요하고, 번들이 무거워 서버리스 콜드 스타트에 불리. 이 프로젝트는 TypeScript 코드 자체가 스키마+타입인 Drizzle이 개발 속도와 번들 크기 모두에서 유리 |
| **Better Auth** | 아래 상세 설명 참고 | NextAuth(가장 인지도 높음), Clerk, Supabase Auth | 아래 상세 설명 참고 |

**Better Auth 선택 이유 상세:**

**① 네이티브 통합이란?** Better Auth는 `drizzleAdapter(db, { schema })`로 이미 프로젝트에서 쓰는 Drizzle 인스턴스와 스키마를 직접 전달한다. 별도 어댑터 패키지 설치나 스키마 변환 없이, 내가 정의한 `authUsers`, `authSessions` 테이블을 그대로 인증에 사용. NextAuth는 자체 스키마 규격이 있어서 Drizzle과 연결하려면 별도 어댑터(`@auth/drizzle-adapter`)를 설치하고 스키마를 맞춰줘야 한다.

**② DB 훅이란?** `databaseHooks`는 인증 관련 DB 작업(사용자 생성, 세션 생성 등) 전후에 커스텀 로직을 끼워넣는 라이프사이클 콜백이다. 이 프로젝트에서 실제 사용하는 훅:
- `user.create.after`: 회원가입 완료 직후 실행
- `session.create.before` / `session.update.before`: 세션 저장 전에 IP·UserAgent 최소화 (개인정보 보호)

**③ 자동 시딩이란?** 신규 사용자가 가입하면 `user.create.after` 훅에서 즉시 기본 카테고리 12개(식비, 교통, 카페/간식 등)를 DB에 삽입한다. 이렇게 하면 사용자가 가입 직후 첫 거래를 바로 입력할 수 있다. 이걸 안 하면? → AI 파서가 "카테고리 없음" 예외를 처리해야 하고, 사용자는 온보딩 단계에서 카테고리를 직접 만들어야 하는 마찰이 생긴다.

**시딩 확장 가능성**: 현재는 한국어 기본 12개 카테고리만 고정 시딩하지만, DB 훅 기반이라 확장이 쉽다:
- **직업/라이프스타일별 프리셋**: 가입 시 "직장인 / 자영업자 / 학생" 선택 → 맞춤 카테고리 시딩 (학생에겐 "등록금", 자영업자에겐 "매출/재료비")
- **기본 계좌 시딩**: 카테고리처럼 "현금", "주거래 은행" 같은 기본 계좌도 자동 생성
- **기본 예산 템플릿**: 카테고리와 함께 권장 예산 비율도 초기 세팅
- 이 모든 확장이 `user.create.after` 훅 한 곳에서 관리되므로 인증 로직과 분리된 채 독립적으로 발전 가능

**NextAuth 장점 (업계 표준인 이유):** npm 주간 200만+ 다운로드, 커뮤니티/레퍼런스 방대, Google·GitHub 등 소셜 로그인 20+ provider 기본 탑재. 이 생태계 규모와 안정성이 표준의 근거. 그러나 DB 훅이 없어서 "가입 → 카테고리 생성"을 인증 레이어에서 처리할 수 없고, 별도 API나 미들웨어를 만들어야 한다.
| **Tailwind CSS 4** | 유틸리티 퍼스트로 빠른 프로토타이핑, oklch 컬러 내장 | CSS Modules, styled-components | 클래스명 길어지지만 디자인 시스템과 궁합 최고 |
| **Capacitor** | 하나의 코드베이스로 웹+iOS+Android 릴리즈, 네이티브 API 접근 가능 | React Native, Expo | 네이티브 성능은 RN 대비 약하지만 코드 공유율 100% |

<summary><h2>3. AI 파이프라인 — 10단계 흐름</h2></summary>

```
사용자 입력 (텍스트/이미지)
    ↓
① 빈 입력 검증
    ↓
② OOD 필터 (가계부 무관 입력 차단)
    ↓
③ 은행 메시지 전처리 (잔액/한도 등 노이즈 제거)
    ↓
④ Provider 결정 (텍스트: 길이 기반 / 이미지: 사용량 기반)
    ↓
⑤ DB 병렬 조회 (카테고리 + 계좌)
    ↓
⑥ 프롬프트 조립 (시스템 프롬프트 + 사용자 데이터 동적 주입)
    ↓
⑦ LLM 호출 (타임아웃 + 폴백)
    ↓
⑧ 응답 검증 + JSON 파싱
    ↓
⑨ 확인 시트 표시 (사용자 수정 가능)
    ↓
⑩ 확정 후 DB 저장 + 계좌 잔액 반영
```

### 3-1. OOD 필터 (Out-of-Domain)

```
목적: LLM 호출 전에 비용/속도 절감
전략: "false negative 최소화" — 애매하면 통과시키고 LLM 2차 필터에 위임

1차 필터 (규칙 기반):
  ✓ 금융 단서 있으면 즉시 통과 (FINANCIAL_HINTS 100+ 키워드)
  ✓ 금액 패턴 (숫자+원) → 통과
  ✓ 은행 메시지 패턴 → 통과
  ✗ 명백한 OOD만 차단 (날씨, 코딩, 번역 등)
  
2차 필터 (LLM 자체):
  → 프롬프트에 "가계부와 무관하면 rejected: true" 지시
```

**면접 답변 포인트**: "정확한 차단보다 놓치지 않는 것이 더 중요합니다. 1건의 잘못된 거래 입력은 사용자가 확인 시트에서 삭제하면 되지만, 정상 입력이 차단되면 사용자는 앱이 고장 났다고 느낍니다."

### 3-2. Provider 라우팅 (3-Provider 전략)

**도입 히스토리** (면접에서 "왜 3개나?" 질문 대비):
1. **Kimi로 시작** — 한국어 파싱 정확도가 가장 높아서 최초 선택
2. **속도 개선 필요** → 호스팅이 다른 **Fireworks** 발견 (Kimi 모델을 자체 인프라에서 서빙, 이미지 무료 티어도 제공)
3. **추가 실험** → **MiniMax**를 테스트 목적으로 추가 (짧은 입력에서 빠른 응답)

> 이 프로젝트는 프로덕션 제품이기도 하지만 **다양한 LLM 벤더를 직접 붙여보고 비교하는 학습/실험 프로젝트**이기도 합니다. 그래서 Provider 추가가 쉬운 아키텍처(팩토리 패턴)를 먼저 설계한 것입니다.

```
텍스트 입력:
  ≤100자 → MiniMax(빠름) → [실패 시] Fireworks → [최후] Kimi
  >100자 → Kimi (복잡한 입력 처리에 강함)

이미지 입력:
  세션당 Fireworks 3회 → [초과 시] Kimi
  장애 시 10분 쿨다운 → 즉시 Kimi로 전환
```

| Provider | 특징 | 모델 | 용도 |
|----------|------|------|------|
| **MiniMax** | 빠른 응답, 짧은 입력 최적화, `reasoning_split` 지원 | MiniMax-M2.5 | 단순 텍스트 (100자 이하) |
| **Fireworks** | Kimi를 서빙하는 인퍼런스 플랫폼, 이미지 무료 티어 | kimi-k2p5 | 이미지 (세션당 3회) |
| **Kimi** | 한국어 파싱 정확도 최고, 긴 입력 처리 | kimi-k2.5 | 긴 텍스트 + 이미지 폴백 |

**핵심 설계 결정**:
- OpenAI SDK 통일 인터페이스: `baseURL`만 바꿔서 3개 벤더를 동일 코드로 호출
- 싱글턴 캐시: 같은 provider는 한 번만 생성하고 재사용 (메모리 절약)
- 성공 응답만 usage 차감: 장애에 의한 할당 소진 방지

### 3-3. 프롬프트 엔지니어링

```
시스템 프롬프트 구조:
  ① 역할 정의 (한국 가계부 AI 어시스턴트)
  ② intent 판별 규칙 (거래 vs 자산)
  ③ 금액/날짜 파싱 한국어 특수 규칙
  ④ 카테고리 매핑 힌트 (사용자별 동적 주입)
  ⑤ 계좌 매핑 힌트 (사용자별 동적 주입)
  ⑥ JSON 출력 스키마 정의
  ⑦ OOD 거부 규칙 + 예제
```

**사용자별 동적 주입**: 프롬프트에 해당 사용자의 카테고리 목록과 계좌 목록을 `DB 조회 → 주입`하여 LLM이 사용자 맞춤 분류를 수행

**① 역할 정의 + 공통 원칙**
```
"당신은 한국어 가계부 입력을 구조화된 JSON으로 변환하는 파서입니다.
 설명문, 인사말, 마크다운 없이 반드시 JSON 객체 하나만 반환하세요."
```
- 오늘 날짜를 `${today}`로 주입 → "어제", "그제", "2/14" 같은 상대 날짜 계산 기준
- 금액은 양의 정수 원 단위: `"9천"=9000`, `"2.3만"=23000`, `"350만원"=3500000`
- description은 상호명/품목 중심 2~30자, 잔액·승인번호·카드번호 같은 노이즈 제외

**② intent 판별 규칙** — 가장 핵심적인 부분
```
transaction (돈의 이동): 결제, 승인, 입금, 출금, 이체, 환불...
account (계정 상태):     잔액, 잔고, 보유 현금, 미결제, 대출 잔액...
```
- "카카오뱅크 잔액 150만" → account (거래 동사 없음)
- "스타벅스 4500" → transaction (소비 맥락)
- 혼합 입력 → 두 배열 모두 채움

**③ type 판별 (income vs expense) — 8개 우선순위 규칙**
1. 환불/캐시백/보험금 → income
2. 급여/용돈/판매대금 → income
3. 결제/구매/택시/월세 → expense
4. **적금 납입/주식 매수 → expense** (함정: 수입이 아님!)
5. "입금"만으로 income 단정 금지 (대출 실행, 계좌 간 이동 가능)
6. "출금/이체"는 대체로 expense, 단순 잔액 보고면 account 우선

**④ 카테고리 매핑 힌트** (사용자별 동적 주입)
```
프롬프트에 주입되는 형태:
  - 지출: 식비, 카페/간식, 교통, 주거/관리비, 생활용품, ...
  - 수입: 급여, 용돈/부수입, 투자수익, 기타 수입
```
- 매핑 힌트: 커피/디저트 → 카페/간식, 월세/관리비 → 주거/관리비 등 14개 계열
- 적절한 카테고리 없으면 `기타 지출/수입` 선택 + `suggestedCategory`로 새 후보 제안

**⑤ 계정(account) 파싱 규칙**
```
asset: 은행예금, 현금, 적금, CMA, 증권계좌...
debt:  카드 미결제, 할부 잔액, 대출, 마이너스통장...
subType: bank | cash | savings | investment | credit_card | loan | other
icon:    🏦  | 💵   | 🏧     | 📈        | 💳          | 🏠  | 📦
```
- 계정명 정제: "카카오뱅크 잔액" → "카카오뱅크" (노이즈 제거)
- 기존 계정과 이름 유사하면 재사용: "카뱅" → "카카오뱅크"

**⑥ JSON 출력 스키마**
```json
{
  "intent": "transaction" | "account",
  "transactions": [{ "date", "type", "category", "description", "amount",
                      "isRecurring", "dayOfMonth", "suggestedCategory" }],
  "accounts": [{ "name", "type", "subType", "icon", "balance" }]
}
```

**⑦ OOD 거부**
```json
{"rejected": true, "reason": "가계부와 관련 없는 입력입니다."}
```
- 거부: 날씨, 코딩, 잡담, 번역
- 거부 금지: 금액 포함 소비/수입/잔액/대출/카드 메시지

<summary><h2>4. 데이터베이스 설계</h2></summary>

### 핵심 테이블 (7개)

| 테이블 | 목적 | 핵심 설계 |
|--------|------|-----------|
| `categories` | 수입/지출 카테고리 | `userId`+`type`+`name` 유니크 인덱스 → 중복 방지 |
| `transactions` | 거래 내역 | `userId`+[date](file:///Users/leeth/Documents/git/Household%20account%20book/src/lib/cache-keys.ts#28-32) 복합 인덱스 → 월별 조회 최적화 |
| `accounts` | 자산/부채 계정 | `balance`는 `text` 타입 → AES-256 암호화 저장 |
| `recurring_transactions` | 고정 거래 | `dayOfMonth`로 매월 자동 생성 |
| `budgets` | 월별 예산 | `userId`+`categoryId`+`month` 유니크 → 중복 방지 |
| `security_rate_limits` | 보안 속도 제한 | `scope`+`keyHash` 유니크 + 점진적 차단 |
| `security_events` | 보안 이벤트 로그 | 감사 추적용 |

### 🔍 테이블 분리 이유 — 면접관 반론 대비

**Q: "거래와 고정거래를 하나의 테이블로 합치면 안 되나요?"**
> `transactions`는 **실제 발생한 거래 기록**(불변 팩트)이고, `recurring_transactions`는 **매월 반복 생성되는 템플릿**(규칙 정의)입니다. 역할이 다릅니다. 합치면 `isRecurring`, `dayOfMonth`, `lastApplied` 같은 필드가 일반 거래 행에는 항상 NULL이 되어 스키마가 지저분해지고, "이번 달 고정지출 목록" 같은 쿼리에서 전체 거래를 스캔해야 합니다. 분리하면 각 테이블이 자기 역할에 최적화된 인덱스를 가질 수 있습니다.

**Q: "카테고리를 테이블로 분리하지 않고 거래에 문자열로 넣으면?"**
> 문자열이면 "식비", "식대", "밥값"이 전부 다른 카테고리가 됩니다. 테이블로 분리하면 FK로 참조 무결성을 강제하고, 카테고리명을 변경해도 과거 거래가 자동 반영됩니다. 또 `type` 필드(income/expense)가 카테고리에 속하므로 거래 생성 시 수입 카테고리를 지출에 할당하는 실수를 DB 레벨에서 방지합니다.

**Q: "보안 테이블(rate_limits, events)은 애플리케이션 로직에서 처리하면 되지 않나요?"**
> 메모리에서 관리하면 서버 재시작 시 초기화됩니다. DB에 저장하면 서버 인스턴스가 여러 개여도 속도 제한이 일관되고, 보안 이벤트 로그는 감사 추적에 필수입니다. 다만 현재는 단일 서버니까 인메모리로도 충분한데, 확장성을 고려해 DB 기반으로 설계했습니다.

**Q: "accounts 테이블의 balance를 왜 text로 저장하나요? numeric이 자연스럽지 않나요?"**
> AES-256-GCM 암호화 결과가 바이너리/Base64 문자열이기 때문입니다. 복호화 후 Number로 변환해서 사용합니다. 잔액은 직접 SQL 집계 대상이 아니라(거래 금액으로 계산), 암호화 우선으로 결정했습니다.

**Q: "budgets을 별도 테이블로 안 하고 categories에 budget 필드를 넣으면?"**
> 예산은 **월별**로 다릅니다. 1월 식비 예산 30만, 2월 식비 예산 40만. 카테고리에 넣으면 이력이 사라집니다. `userId`+`categoryId`+`month` 유니크 제약으로 같은 카테고리의 같은 달 중복 예산도 방지합니다.

### 거래-계좌 잔액 연동 로직

```
거래 생성 시:
  수입(income) → 계좌 잔액 + amount
  지출(expense) → 계좌 잔액 - amount
  
거래 삭제 시:
  → 역산: 원래 적용의 반대 방향으로 잔액 조정
  
거래 수정 시:
  → 이전 금액 역산 + 새 금액 적용
```

### 암호화 전략 (4계층 모델)

| 계층 | 이름 | 무엇을 보호? | 누가 제공? |
|:---:|------|------------|-----------|
| **①** | **Encryption at Rest** (저장 시 암호화) | 디스크에 저장된 모든 데이터. DB 서버가 물리적으로 탈취당해도 평문 노출 방지 | Supabase/PostgreSQL 자동 적용 (AES-256) |
| **②** | **TLS** (전송 암호화) | 네트워크 구간 (클라이언트↔서버, 서버↔DB). 중간자 공격 방지 | HTTPS + Supabase SSL 연결 |
| **③** | **접근 제어** (RLS + 세션) | 인증된 사용자만 **자기 데이터**에 접근. 타 사용자 데이터 열람 차단 | Better Auth 세션 + 모든 쿼리에 `userId` 필터 |
| **④** | **앱 레벨 암호화** (AES-256-GCM) | DB 관리자(DBA)나 DB 접근 권한이 유출되어도 민감 필드는 복호화 키 없이 읽을 수 없음 | 직접 구현 (`crypto.ts`) — **선택적 적용** |

> ①②③은 **모든 필드에 자동 적용**. ④는 SQL 쿼리가 불필요한 민감 필드에만 **추가 적용**.

**Q: "① at Rest가 있는데 ④ 앱 레벨 암호화가 필요한 이유는?"**

> Encryption at Rest는 **디스크가 꺼져 있을 때만** 보호합니다. DB 프로세스가 실행 중이면 자동 복호화돼서 쿼리 결과는 평문입니다. 즉 **DB 접속 권한만 얻으면 다 보입니다**.
>
> ※ SQL Injection은? → Drizzle ORM이 모든 쿼리를 **파라미터화 쿼리**로 실행하므로 원천 차단됨. 아래는 DB 접속 권한 자체가 유출되는 시나리오:

| 공격 시나리오 | ①②③만 | ①②③ + ④ |
|-------------|:---:|:---:|
| 서버실 침입 → 디스크 탈취 | ✅ | ✅ |
| **[.env](file:///Users/leeth/Documents/git/Household%20account%20book/.env) 유출 → DB 커넥션 스트링 노출** | ❌ 평문 노출 | ✅ 암호문만 보임 |
| **Supabase 대시보드 계정 해킹** | ❌ 평문 노출 | ✅ 암호문만 보임 |
| **DB 백업 파일이 외부로 유출** | ❌ 평문 노출 | ✅ 암호문만 보임 |
| **내부자(DBA)가 직접 조회** | ❌ 평문 노출 | ✅ 암호문만 보임 |

> 예: [.env](file:///Users/leeth/Documents/git/Household%20account%20book/.env) 파일이 실수로 GitHub에 푸시되어 DB 접속 정보가 노출돼도, ④가 적용된 필드는 앱 서버의 `ENCRYPTION_KEY` 없이는 읽을 수 없습니다.

아래 표는 ④번 앱 레벨 암호화를 **추가 적용했는지** 여부:

| ④ 적용 ✅ | ④ 미적용 (①②③으로 보호) | 미적용 이유 |
|-----------|---------------------|-----------|
| 계좌명 (`name`) | 금액 (`amount`) | `SUM`, `GROUP BY` 등 SQL 집계 필수 |
| 잔액 (`balance`) | 설명 (`description`) | 텍스트 검색 필요 |
| 원본 입력 (`originalInput`) | 날짜 ([date](file:///Users/leeth/Documents/git/Household%20account%20book/src/lib/cache-keys.ts#28-32)) | 범위 조회 (`WHERE date BETWEEN`) |
| 메모 (`memo`) | 타입 (`type`) | 필터링 (`WHERE type = 'expense'`) |

**Q: "금액도 ④를 적용할 수는 없나요?"**

> 가능하지만, 그러면 SQL 집계가 불가능해져서 **앱 서버에서 전체 행을 가져와 복호화 후 메모리에서 집계**해야 합니다. 거래가 수천 건이면 성능 문제가 생깁니다. 실제 은행/카드사도 같은 트레이드오프를 합니다 — 집계·리포팅 대상 컬럼은 DB 레벨 암호화 + 접근 제어로 보호하고, 앱 레벨 암호화는 집계 불필요 필드에만 적용합니다.
>
> **개선 계획**: 거래 수가 많아지면 ①앱 레벨 복호화+메모리 집계 또는 ②PostgreSQL `pgcrypto` 확장으로 DB 내 복호화 집계를 도입할 수 있습니다.

**Q: "금액이 평문이면 SUM으로 잔액 역산 가능하지 않나요? 잔액 암호화가 의미 있나요?"**

> **솔직히 맞습니다.** 거래 금액을 합산하면 잔액을 추론할 수 있어, 잔액 암호화의 보호 효과는 제한적입니다. 다만 완전히 무의미하지는 않습니다:
>
> - **초기 잔액 보호**: 계좌 생성 시의 시작 잔액은 거래 기록에 없으므로, 거래만으로는 정확한 현재 잔액을 알 수 없습니다
> - **공격 비용 증가**: `SELECT balance` 한 줄 vs 전체 거래 조회·합산 → 즉석 조회(quick glance) 방지
>
> 하지만 이건 **완전한 방어가 아닌 지연·방해 수준**입니다. 솔직하게 인정하되, 왜 이 수준에서 타협했는지를 설명하는 것이 면접에서 더 신뢰를 줍니다:
>
> "금액까지 ④를 적용하면 잔액 역산이 불가능해지지만, SQL 집계가 막히는 트레이드오프가 있습니다. 현재 MVP에서는 이 수준의 타협을 택했고, 향후 앱 레벨 복호화+메모리 집계로 전환하면 금액과 잔액 모두 ④ 적용이 가능합니다."

<summary><h2>5. Next.js 아키텍처 결정</h2></summary>

### Server Components vs Client Components

```
Server Components (기본값):
  - 페이지 레벨 데이터 로딩 (DB 직접 접근)
  - API 키 노출 없이 LLM 호출
  - 번들 크기 기여 0

Client Components (명시적 "use client"):
  - 이벤트 핸들러 (onClick, onChange)
  - useState/useEffect 필요 시
  - 차트 라이브러리 (Recharts)
```

### Server Actions vs API Routes

| | Server Actions | API Routes |
|--|---------------|------------|
| **용도** | CRUD 작업 | 복잡한 파싱 로직 |
| **CSRF** | 자동 관리 | 수동 검증 필요 |
| **예시** | 거래 저장/수정/삭제, 카테고리 관리 | `/api/parse` (이미지 업로드, 취소, HTTP 상태 제어) |
| **캐시** | `revalidatePath()` 호출 | 직접 관리 |

**왜 파싱만 API Route?**
→ 이미지 업로드(`multipart/form-data`), `AbortController` 지원, HTTP 상태 코드 세밀 제어가 필요. Server Actions는 이런 저수준 제어에 부적합.

### 전역 상태 관리를 안 쓰는 이유

"React Server Components 환경에서는 서버에서 직접 데이터를 가져오므로, 클라이언트 상태 저장소(Redux, Zustand 등)의 '서버 데이터 캐싱' 역할이 불필요합니다. URL(검색 파라미터)이 상태 역할을 하고, `revalidatePath`가 캐시 무효화를 처리합니다."

**`revalidatePath`란?** Next.js의 서버 캐시 무효화 함수다. RSC는 페이지 데이터를 서버에서 캐시하는데, 데이터가 변경되면 이 캐시를 날려야 최신 데이터가 표시된다. `revalidatePath("/transactions", "page")`를 호출하면 거래 페이지의 캐시가 즉시 만료되어 다음 접근 시 DB에서 새로 가져온다.

**이 프로젝트에서의 실제 적용** ([cache-keys.ts](file:///Users/leeth/Documents/git/Household%20account%20book/src/lib/cache-keys.ts)): mutation 종류별로 연관된 페이지들을 일괄 무효화하는 헬퍼 함수를 만들었다:
```
거래 생성/수정/삭제 → revalidateTransactionPages()
  → /transactions + /statistics + /assets + /budget 4개 페이지 캐시 날림

계좌 변경 → revalidateAccountPages()  → /assets
카테고리 변경 → revalidateCategoryPages() → 5개 페이지
```
이렇게 하면 거래를 저장하면 관련된 통계/자산/예산 페이지도 자동으로 최신화된다. 클라이언트에서 `fetch` 후 `setState`로 동기화할 필요가 없다.

|--------|------|
| **인증** | Better Auth + 쿠키 기반 세션 |

| **속도 제한** | `securityRateLimits` 테이블 + 점진적 차단 (아래 상세) |
| **데이터 암호화** | AES-256-GCM (계좌명, 잔액, 원본입력, 메모) |
| **API 보호** | API Route에서 세션 검증 + 환경변수로 키 관리 |
| **미들웨어** | `/login`, `/register` 제외 모든 경로 보호 |

### 속도 제한 구현 상세 ([consumeRateLimit](file:///Users/leeth/Documents/git/Household%20account%20book/src/server/security/index.ts#135-289))

**"윈도우(window)"란?** 요청 횟수를 세는 시간 구간이다. 예를 들어 "10분 윈도우에 max 10회"면 → 10분 동안 10번까지 허용, 10분 지나면 카운터 리셋. `windowStartedAt`이 시작 시각, `windowSeconds`가 구간 길이.

```
consumerRateLimit(요청) → DB 트랜잭션 내에서:

1️⃣ scope+keyHash로 행 조회 (SELECT ... FOR UPDATE → 동시 요청 직렬화)
2️⃣ 차단 중이면? → blockedUntil 확인 → 고착 거부 + retryAfterSeconds 반환
3️⃣ 윈도우 만료? → 카운터 초기화 (requestCount=1), 허용
4️⃣ 카운터 < max? → requestCount++ → 허용
5️⃣ 카운터 >= max? → 차단 → 점진적 에스컀레이션
```

**점진적 에스컀레이션이란?**
- `consecutiveBlocks` 카운터를 누적
- 3회 이상 연속 차단(`escalateAfter`) → 차단 시간이 15분(`blockSeconds`)으로 연장
- 3회 미만이면 윈도우 만료까지만 대기
- 예: 1차 차단=윈도우 만료까지, 2차=똑같이, 3차=15분 차단!

**실제 사용 예시** ([consumeIpAnomalyLimit](file:///Users/leeth/Documents/git/Household%20account%20book/src/server/security/index.ts#305-349)):
```
비정상 입력 감지(Base64 위장, CSRF 위반 등) 시:
  scope: "anomaly:parse"
  max: 10회 / 10분 윈도우
  3회 연속 초과 → 15분 차단
  + securityEvents 테이블에 이벤트 기록 (감사 추적)
```

**면접 답변 포인트**: "단순 회수 제한이 아니라, 반복 위반 시 차단 시간이 점점 늘어나는 점진적 페널티 시스템입니다. DB 트랜잭션 + `SELECT FOR UPDATE`로 동시 요청 경합 상황도 안전하게 처리합니다."

### 디자인 시스템

- **컬러**: oklch 기반 그린 모노톤 → 수학적으로 일관된 명암 단계
  - **oklch란?** `oklch(밝기, 채도, 색상각도)`로 색을 정의하는 최신 컬러 스페이스. 기존 `hsl`은 밝기(Lightness)가 인간 눈의 밝기 인식과 불일치 — 예를 들어 `hsl(120,100%,50%)` 녹색과 `hsl(240,100%,50%)` 파랑은 같은 L이지만 눈에는 밝기가 다르다. oklch는 **지각적으로 균일**해서 L 값만 바꾸면 자연스러운 명암 스케일이 나온다. Tailwind CSS 4가 기본 지원.
  - **왜 가계부에?** primary-50~900까지 10단계 컬러를 생성할 때, oklch면 L 값만 일정 간격으로 조절하면 되어 "디자이너 없이도" 일관된 컬러 시스템을 만들 수 있다.
- **수입/지출**: 파란색(수입) / 빨간색(지출) → 문화적 직관
- **타이포그래피**: Pretendard (한글) + Geist Sans (영문/숫자)
  - **Pretendard 선정 이유**: ①Apple SD Gothic Neo 대체 목적으로 설계된 한글 전용 고딕체, ②system-ui 급 가독성이면서 무료 웹폰트, ③한국 IT 업계 사실상 표준 (토스, 카카오, 네이버 등 채택). 가계부는 숫자와 한글이 섞여야 해서 한글 글리프 품질이 핵심.
  - **Geist Sans 선정 이유**: Vercel이 만든 Next.js 공식 폰트. ①숫자·통화 표시에 최적화된 고정폭 숫자(tabular figures) 지원, ②`next/font`로 최적 로딩 내장, ③Pretendard와 x-height가 비슷해 혼용 시 시각적 통일감 유지.
- **모바일 퍼스트**: 최소 터치 영역 44px 보장, 카드 기반 레이아웃
  - **왜 44px?** Apple Human Interface Guidelines(HIG)에서 정의한 최소 터치 타깃 크기. 성인 손가락 끝 평균 접촉 면적이 약 44×44pt. 이보다 작으면 오터치율이 급증한다. Google Material Design도 48dp를 권장하는데, 44px은 실질적 업계 최소 기준. 가계부는 금액·날짜 등 빈번한 터치가 필요해서 이 기준을 엄격 적용.

</details>

<summary><h2>10. 예상 질문 TOP 10 + 모범 답변</h2></summary>

### Q1. "이 프로젝트를 한 줄로 소개해 주세요"
> "자연어 입력만으로 AI가 수입/지출을 자동 분류해주는 개인 가계부입니다. 가계부 입력이 귀찮아서 포기하는 사람들의 문제를 해결합니다."

### Q2. "AI 파싱 정확도가 낮으면 어떻게 대응하나요?"
> "3중 안전장치입니다. ①OOD 필터로 무관 입력 사전 차단, ②LLM 자체의 `rejected` 필드로 2차 검증, ③확인 시트에서 사용자가 최종 수정. 원본 입력을 항상 보존해서, 향후 수정 패턴을 분석해 프롬프트를 개선할 수 있습니다."

### Q3. "왜 3개 Provider를 쓰나요? 하나로 통일하면 안 되나요?"
> "비용, 속도, 안정성의 균형입니다. MiniMax는 짧은 입력에 빠르고, Fireworks는 이미지 무료 티어가 있고, Kimi는 한국어 정확도가 높습니다. 하나가 장애 나도 자동 폴백으로 서비스가 계속됩니다. 설계적으로는 OpenAI SDK의 `baseURL`만 바꾸는 팩토리 패턴으로 Provider 추가가 설정 파일 한 줄이면 끝납니다."

### Q4. "Drizzle ORM을 선택한 이유는?"
> "Prisma가 ORM 생태계 1위이고 마이그레이션 자동화, Prisma Studio 같은 GUI, 풍부한 문서/커뮤니티가 장점인 건 알고 있습니다. 그럼에도 Drizzle을 택한 이유는 ①번들 크기가 작아 서버리스 콜드스타트에 유리, ②SQL에 가까운 API라 `.prisma` 스키마 → `generate` 단계 없이 TS 코드가 곧 스키마+타입, ③복잡 쿼리도 SQL 직관대로 작성 가능하기 때문입니다."

### Q5. "Server Actions와 API Routes를 어떻게 구분하나요?"
> "데이터 변경(CRUD)은 Server Actions, 복잡한 IO 제어가 필요한 것은 API Routes입니다. 파싱 API만 Route로 분리했는데, 이미지 `multipart/form-data` 처리, `AbortController` 기반 취소, HTTP 상태 코드 세밀 제어가 필요했기 때문입니다."

### Q6. "전역 상태 관리(Redux 등)를 안 쓰는 이유는?"
> "의도적 결정입니다. RSC 환경에서 서버가 데이터 소스이므로, 클라이언트에 상태를 복제할 필요가 없습니다. URL 파라미터가 상태 역할을 하고, `revalidatePath`가 캐시를 무효화합니다. 상태 동기화 버그를 원천 차단합니다."
>
> "그래도 쓰면 장점?" → ①낙관적 업데이트(서버 응답 전에 UI 즉시 반영)가 쉬워지고, ②클라이언트 간 실시간 상태 공유(웹소켓 + 스토어), ③오프라인 모드 구현이 가능합니다. 현재 가계부는 이런 요구사항이 없어서 도입하지 않았지만, 향후 가족 공유 가계부 같은 기능이 생기면 Zustand 도입을 검토할 수 있습니다."

### Q7. "AES-256 암호화를 어떤 기준으로 적용했나요?"
> "우선 모든 데이터는 Supabase의 Encryption at Rest(AES-256, 기본 활성화 — 팩트 확인 완료)로 디스크 레벨에서 보호됩니다. 거기에 **앱 레벨 암호화**(AES-256-GCM)를 추가 적용한 기준은 '① 민감도가 높고 ② SQL 쿼리가 불필요한 필드'입니다. 계좌명·잔액·원본입력·메모가 해당됩니다. 금액은 집계 필수라 앱 레벨 미적용이지만, DB 커넥션 유출 시나리오까지 방어하려면 향후 앱 레벨 복호화+메모리 집계로 전환할 수 있습니다."

### Q8. "가장 어려웠던 기술적 문제는?"
> "백엔드 설계, 특히 DB 스키마 설계입니다. 저는 원래 프론트엔드 개발자여서 테이블 분리 기준, 관계 설정, 인덱싱 같은 결정이 처음이었습니다. AI에게 '해줘'가 아니라 **계속 대화하고 토론**했습니다 — '이 테이블은 왜 분리하는지', '이 관계가 나중에 어떤 문제를 일으킬 수 있는지', 장단점과 제 프로젝트 목표를 공유하면서 설계를 함께 만들어갔습니다. 이 과정에서 정규화, 트레이드오프, 보안 계층 같은 개념을 실전으로 익혔고, 결과적으로 7개 테이블 구조를 직접 이해하고 설명할 수 있게 됐습니다."

### Q9. "바이브 코딩(AI 활용 개발) 어떻게 했나요?"
> "Q8에서 말씀드린 것처럼, AI에게 '만들어줘'가 아니라 **토론 상대**로 활용했습니다. 예를 들어 DB 스키마를 설계할 때 'transactions와 recurringTransactions를 왜 분리해야 하지?', '카테고리를 문자열로 하면 어떤 문제가 생기지?' 같은 질문을 던지고, AI의 답변에 '그런데 이 경우엔 어떻게 되냐' 식으로 반론하면서 설계를 다듬었습니다. 보안 계층도 마찬가지로 — 'Encryption at Rest만으로 충분하지 않냐'고 물으면 AI가 시나리오를 제시하고, 제가 우리 프로젝트에 맞는 수준을 판단하는 방식이었습니다. 코드 자체는 AI가 빠르게 작성하지만, **무엇을 왜 이렇게 설계하는지는 제가 이해하고 결정**했습니다. 그래서 지금 이 면접에서 모든 설계 결정을 직접 설명할 수 있는 겁니다."

### Q10. "1인 프로젝트에서 배운 것은?"
> "프론트엔드만 해오다가 백엔드·DB·보안·인프라·LLM까지 혼자 결정해야 했는데, 가장 크게 배운 건 **'정답은 없고 트레이드오프만 있다'**는 것입니다. 금액 암호화 여부, Provider 몇 개를 쓸지, 전역 상태 관리를 도입할지 — 모두 '이상적인 답'이 아니라 '이 프로젝트의 현재 단계에서 가장 합리적인 타협'을 찾는 과정이었습니다. 그리고 그 판단 근거를 설명할 수 있어야 의미가 있다는 것도 배웠습니다. 이 프로젝트가 포트폴리오인 동시에 **개인 학습 기록**이기도 한 이유입니다."

</details>
