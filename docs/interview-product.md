# 🎯 제품 이해 면접 대응 매뉴얼

> 36개 질문을 5개 카테고리로 정리. 각 답변은 **"왜 그렇게 결정했나"**에 초점.

---

## Part A. 제품 이해 (8문)

### Q1. 이 앱이 해결하려는 가장 큰 사용자 문제를 한 문장으로?

> **가계부를 쓰고 싶지만 "입력이 귀찮아서" 3일 만에 포기하는 문제.**

기존 앱은 날짜 선택 → 카테고리 선택 → 금액 입력 → 메모 입력까지 4단계를 거쳐야 한다. 이 앱은 "점심 김치찌개 9000"이라는 한 문장으로 4단계를 한 번에 끝낸다.

### Q2. 왜 "가계부를 쉽게 쓰게 하는 것"보다 "AI 자동 분류"를 전면에 둔 제품이라고 판단했나?

"쉽게 쓰게 하는 것"은 목표이고, "AI 자동 분류"는 그 목표를 달성하는 **수단**이다. 전면에 둔 이유:
- **차별점이 명확**: 기존 뱅크샐러드, 토스 가계부도 "쉽게 쓰게" 하지만, 자연어 → AI 분류는 이 앱만의 체험. 사용자가 처음 써볼 때 "와, 이거 되네?" 하는 순간을 만들어야 리텐션으로 이어진다.
- **입력 장벽이 핵심 이탈 원인**: 가계부 앱 이탈률 1위가 "입력 귀찮음". AI가 분류/날짜/금액을 자동 처리하면 이 장벽이 사라진다.

### Q3. 핵심 타깃 사용자는 누구이며, 왜 특히 유효한가?

**20~30대 직장인 중 가계부를 써본 적 있지만 포기한 사람.**
- 소비를 관리해야 한다는 인식은 있지만 **행동으로 옮기기 귀찮은** 사람
- 카카오톡/SMS로 결제 알림을 받는 사람 (은행 메시지 붙여넣기 기능의 직접적 타깃)
- 복잡한 투자 관리보다 **일상 소비 추적**이 목적인 사람
- 이 그룹에 특히 유효한 이유: "입력 0초"에 가까울수록 지속률이 올라가고, 이 앱의 자연어+은행 메시지 파싱이 정확히 그 지점을 공략

### Q4. 개인 가계부로 잡은 이유?

- **MVP 범위 제한**: 공유 가계부는 "누가 뭘 입력했나", "결재 승인", "카테고리 합의" 같은 **협업 레이어**가 필요. 이는 인증/권한/알림의 복잡도를 3배 이상 올린다.
- **타깃 문제가 개인적**: "내가 이번 달 어디서 돈을 많이 썼지?"는 개인적 질문. 커플/가족은 이 질문 위에 "정산", "예산 협의" 등 추가 문제가 겹친다.
- **확장 경로**: 개인 가계부를 완성한 뒤 "공유 워크스페이스" 기능을 추가하는 것이 자연스러운 확장. 반대는 어렵다.

### Q5. 사용자가 처음 켰을 때 체감해야 하는 가치?

> **"말로 쓰면 끝나네."**

첫 화면에서 입력바에 "점심 비빔밥 8000"을 치면, 2~3초 뒤 **카테고리(식비)·금액(8,000원)·날짜(오늘)가 자동으로 채워진 확인 시트**가 올라온다. 이 순간 "진짜 이게 되는구나"를 느껴야 한다. 만약 이 첫 경험이 느리거나(5초+) 틀리면(카테고리 오분류) 앱을 다시 열지 않는다.

### Q6. 기존 가계부 앱 대비 기능적 차별점 2가지?

1. **자연어 AI 자동 분류**: "어제 스벅 4500, 점심 9000" → 2건 거래가 카테고리·날짜·금액까지 자동 파싱. 기존 앱은 건별 수동 입력이거나, 은행 API 연동(은행 앱 인증 필요)에 의존.
2. **은행 알림 메시지 붙여넣기**: 카카오톡/SMS로 받은 "[카카오뱅크] 출금 5,500원 스타벅스"를 복사+붙여넣기하면 자동 파싱. 별도 연동 없이 즉시 사용 가능.

### Q7. "좋은 사용자 경험"을 판단하는 기준?

| 기준 | 구체적 목표 |
|------|-----------|
| **입력 시간** | 1건 거래 기록에 10초 이내 (입력 → 확인 → 저장) |
| **파싱 정확도** | 카테고리·금액·날짜 모두 정확한 비율 80%+ |
| **오류 복구 비용** | AI가 틀려도 1탭으로 수정 가능 (편집 시트) |
| **학습 비용 제로** | 별도 튜토리얼 없이 즉시 사용 가능 |
| **시각적 피드백 즉시** | 저장 후 즉시 달력·차트·목록에 반영 |

### Q8. 이 앱이 잘 작동한다고 볼 수 있는 핵심 지표?

| 지표 | 이유 |
|------|------|
| **D7 리텐션** | 가계부의 핵심 성공 = 7일 이상 계속 쓰는가 |
| **AI 파싱 후 수정률** | 낮을수록 AI 정확도 높음. 30% 이하가 목표 |
| **월 활성 거래 수** | 1인당 월 30건+(일 1건) 이상이면 습관화 성공 |
| **확인 시트 → 저장 전환율** | AI 결과를 보고 실제 저장하는 비율. 90%+ 목표 |

## Part B. 핵심 입력 플로우 (8문)

### Q9. "오늘 점심 9000원, 커피 4500원" 입력 시 기대 경험?

1. 하단 입력바에 텍스트를 입력하고 전송
2. 2~3초 로딩 후 **Bottom Sheet가 올라옴** — 2건 거래가 각각 카드 형태로 표시
3. 각 카드에 **날짜(오늘)·금액(9,000/4,500)·카테고리(식비/카페)·설명(점심/커피)**이 자동 채워짐
4. 사용자는 맞으면 "저장", 틀리면 카드를 눌러 수정, 필요 없으면 X로 삭제
5. 저장 후 → 거래 목록으로 이동 + "거래 저장이 완료됐어요" 배너 표시

### Q10. 여러 거래를 한 번에 나눠서 보여주는 기능이 왜 중요한가?

**현실의 입력 패턴을 반영**: 사용자는 하루 끝에 "오늘 쓴 거 다 기록하자"고 생각하고, 한 번에 3~5건을 입력한다. 건별로 따로 입력하면 3~5번 반복 → 귀찮음 → 포기. 한 문장으로 여러 건을 처리하면 **입력 세션 자체의 수를 줄여** 이탈을 방지한다.

### Q11. AI 결과를 바로 저장하지 않고 확인+수정하게 한 이유?

- **AI는 틀릴 수 있다**: 100% 정확하지 않은 AI 결과를 바로 저장하면 **잘못된 데이터가 쌓인다** → 통계/예산이 무의미해짐 → 앱 전체에 대한 신뢰 상실
- **수정 가능 = 통제감**: 사용자가 "내 데이터는 내가 최종 결정"한다는 통제감을 느끼면 AI 오류에 대한 관용이 높아진다
- **학습 기회**: 어떤 필드를 많이 수정하는지 추적하면 향후 프롬프트/모델 개선의 데이터가 된다

### Q12. "새 카테고리 추가" 흐름을 넣은 이유?

기본 12개 카테고리(식비, 교통, 카페...)로는 모든 사용자의 소비 패턴을 커버할 수 없다.
- "반려동물", "자기개발", "운동" 같은 개인적 카테고리 필요
- LLM이 기존 카테고리 중 적합한 것이 없으면 `suggestedCategory`를 반환 → UI에서 "이 카테고리를 추가할까요?" 제안
- **가이드된 확장**: 사용자가 설정 페이지에서 수동으로 추가할 수도 있지만, **입력 흐름 중에 자연스럽게** 추가하는 것이 마찰을 줄인다

### Q13. 상대 날짜 / 한국어 금액 / 수입·지출 자동 판단 중 영향이 가장 큰 것?

**수입·지출 자동 판단**이 가장 크다.
- 상대 날짜("어제"): 편의성. 틀려도 수정 쉬움.
- 한국어 금액("만원"): 편의성. 이미 대부분 숫자로 입력.
- **수입·지출**: 이걸 틀리면 **부호가 뒤집힘** → 통계가 완전히 반대로 나옴. "월급 300만원"을 지출로 잡으면 이번 달 350만 적자로 표시. 사용자 신뢰에 치명적.

### Q14. 사용자가 자주 수정한다면 실패인가 학습 기회인가?

**수정 유형에 따라 다르다.**

| 수정 유형 | 판단 |
|-----------|------|
| **카테고리 변경** (식비→간식) | ⚠️ 프롬프트 개선 필요 (학습 기회) |
| **금액 수정** | 🔴 실패 (금액은 틀리면 안 됨) |
| **날짜 변경** | 🟡 자연어 해석 개선 필요 |
| **설명 보충** ("점심" → "회사 근처 김밥천국") | ✅ 정상 (사용자의 자발적 보강) |
| **수입↔지출 변경** | 🔴 심각한 실패 |

카테고리 수정은 "부드러운 실패". 금액/타입 수정은 "치명적 실패".

### Q15. AI가 틀렸을 때 신뢰를 유지하는 보완 UX?

1. **즉시 수정 가능**: 확인 시트에서 모든 필드를 탭 한 번으로 편집
2. **"삭제" 명확**: 각 거래 카드에 X 버튼 → 틀린 건을 쉽게 제거
3. **원본 보존**: `originalInput`을 암호화 저장 → "원래 뭘 입력했는지" 추적 가능
4. **후속 수정**: 거래 목록에서도 편집 가능 (저장 후에도 수정 가능)
5. **개선 예정**: 수정 패턴을 수집해서 프롬프트에 반영 (Phase 17)

### Q16. 이미지 파싱까지 확장한 이유?

- **영수증/카드 명세서**: 텍스트로 치기 귀찮은 다건 목록을 사진 한 장으로 처리
- **은행 앱 스크린샷**: 은행 앱에서 "SMS 복사"가 안 되는 경우 스크린샷으로 대체
- **접근성**: 타이핑이 불편한 상황(이동 중, 한 손)에서 카메라/갤러리로 즉시 입력
- **현재 한계**: 클라이언트 이미지 압축 미구현 → 큰 이미지 업로드 시 느림. 개선 필요.

## Part C. 은행 메시지와 자동화 (6문)

### Q17. 은행 알림 붙여넣기는 어떤 사용 순간을 해결하나?

> **"결제 알림이 왔는데, 이걸 그대로 가계부에 넣고 싶다."**

카카오톡으로 "[카카오뱅크] 출금 5,500원 스타벅스 잔액 150,000원"이 왔을 때, 사용자는:
- 기존: 가계부 앱 열기 → 금액 5500 입력 → 카테고리 선택 → 저장 (4단계)
- 이 앱: 메시지 복사 → 붙여넣기 → 확인 → 저장 (2단계)

**핵심 순간**: 결제 알림을 받은 **직후 10초 이내**에 기록을 완료하는 것.

### Q18. 사용자가 가장 기대하는 것: 정확성, 속도, 편의성?

**편의성 > 정확성 > 속도** 순서.
- **편의성**: "복사+붙여넣기만 하면 된다"는 것 자체가 핵심 가치. 수동 입력 대비 마찰 80% 감소.
- **정확성**: 금액이 틀리면 치명적. 하지만 은행 메시지의 금액은 명확한 패턴이므로 정확도 95%+.
- **속도**: 2~5초면 충분히 빠름. 사용자는 "즉시"를 기대하지 "0.5초"를 기대하지 않음.

### Q19. 다건 메시지는 어떤 사용자에게 유용한가?

**하루 치 결제 알림을 한 번에 기록하는 사용자.** 퇴근 후 카카오톡에 쌓인 5~10개 결제 알림을 한꺼번에 복사해서 붙여넣으면, 5~10건 거래가 한 번에 파싱된다. 건별로 복사+붙여넣기를 반복하는 것보다 **N배 빠름**.

### Q20. 잔액/한도/누적 중 어떤 정보만 남겨야 하나?

**결제 금액과 상호명만 남기고 나머지는 제거.**
- 잔액: 가계부에 이미 자산 관리 기능이 있음. 잔액은 계좌 페이지에서 관리.
- 한도: 신용카드 한도는 거래 기록과 무관
- 누적: 신용카드 월 누적 사용액도 거래 기록과 무관
- 할부: "일시불" / "할부 3개월" 정보는 노이즈. 거래 기록에는 총 결제 금액만 필요.

실제 코드(`bank-message.ts:9-17`):
```
잔액 NN원, 누적 NN원, 남은 한도 NN원, 일시불, 할부 N개월 → 모두 제거
```

### Q21. 중복 등록 방어 수준은?

| 방어 수준 | 자동/수동 | 현재 구현 |
|-----------|----------|----------|
| **고정 거래 중복** | 자동 차단 | ✅ 시그니처 매칭(±1일) |
| **같은 메시지 재입력** | ❌ 미구현 | 사용자가 같은 알림을 두 번 붙여넣으면 중복 생성됨 |
| **비슷한 금액 다른 거래** | 수동 | 사용자에게 맡김 (확인 시트에서 삭제) |

**원칙**: "진짜 중복"만 자동으로 막고, "비슷해 보이지만 다른 거래"는 사용자에게 맡긴다. 자동 차단을 공격적으로 하면 정당한 거래(같은 가게에서 하루 두 번 결제)가 누락된다.

**개선 방향**: "이미 비슷한 거래가 있어요. 중복인가요?" 확인 UI 추가.

### Q22. 고정 거래 자연어 등록이 줄이는 반복 불편?

매달 반복되는 거래(월세 60만, 넷플릭스 17,000, 교통카드 5만)를 **처음 한 번만 입력**하면 매월 1일에 자동 생성된다.
- 기존 방식: 매달 첫날에 3~5건을 수동 입력 → 26초 × 5 = 2분
- 이 앱: 자동 적용 (fire-and-forget) → 0초
- **트리거**: 거래 페이지 로드 시 `autoApplyRecurringTransactions()` 실행 → 이번 달 미적용 고정거래를 일괄 생성

## Part D. 조회와 인사이트 (6문)

### Q23. 날짜별 그룹으로 보여주는 이유?

- **시간 순서 = 인간의 기억 구조**: "어제 뭐 먹었지?"는 기억하지만 "카테고리 순서"로는 기억 못 함
- **하루 단위 소비 감각**: "오늘 3만원 썼네" → 날짜 헤더에 **일별 합계**를 표시해서 하루 소비 총액을 즉시 파악
- **구분선 역할**: 100건 목록을 스크롤할 때 날짜 헤더가 **시각적 앵커** 역할

### Q24. 도넛 차트 / 최근 7일 차트 / 달력 뷰를 함께 둔 이유?

| 차트 | 질문에 답함 | 시간 범위 |
|------|-----------|----------|
| **도넛 차트** | "어디에 많이 썼지?" (카테고리 비율) | 이번 달 전체 |
| **최근 7일 바 차트** | "요즘 하루에 얼마나 쓰지?" (일별 추이) | 최근 7일 |
| **달력 뷰** | "언제 많이 썼지?" (날짜별 분포) | 이번 달 전체 |

**3가지를 함께 둔 이유**: 각각 "What/How much/When"이라는 다른 질문에 답한다. 하나로 합치면 정보 밀도가 떨어진다.

### Q25. 대시보드와 통계 페이지의 역할 구분?

| | 거래 페이지(대시보드) | 통계 페이지 |
|-|---------------------|-----------|
| **목적** | 오늘/이번 달의 현황 빠르게 파악 | 장기 추세 분석 |
| **시간 범위** | 이번 달 | 최근 6개월 |
| **차트** | 도넛, 7일 바, 달력 | 6개월 트렌드 바, 카테고리 랭킹 |
| **행동 유도** | 거래 입력/수정 | 소비 패턴 인식 → 예산 조정 |
| **사용 빈도** | 매일 | 월 1~2회 |

### Q26. 카테고리 차트 → 상세 분석 이동 흐름이 필요한 이유?

**"식비가 45만원이야"만으로는 행동이 안 바뀐다.** "식비 45만원 중 배달이 30만원"까지 보여야 "배달을 줄이자"는 행동으로 이어진다.
- 도넛 차트 카테고리 클릭 → 통계 페이지의 해당 카테고리 필터링된 상세 뷰로 이동
- `searchParams.category`로 URL에 카테고리 ID를 전달

### Q27. 검색/필터 추가 후 사용자 행동 변화?

- **Before**: 거래 목록을 위아래로 스크롤하며 "눈으로" 찾음 → 30건 이상이면 포기
- **After**: "스벅"으로 검색 → 이번 달 스벅 소비 총액 즉시 확인 → "이번 달 스벅만 2만원이네"
- **추가 가치**: 필터링된 결과의 **합계**를 보여줌으로써 "카테고리 내 세부 소비"를 추적 가능

### Q28. "어디서 돈을 많이 썼는지" 가장 빨리 이해하는 화면?

**거래 페이지의 카테고리 도넛 차트.** 페이지를 열면 월 요약(수입/지출/잔액) 바로 아래에 도넛 차트가 있다. **시선 이동 없이** 가장 큰 조각 = 가장 많이 쓴 카테고리를 즉시 인식. 숫자보다 시각적 비율이 직관적이다.

## Part E. 예산과 자산 (5문)

### Q29. 예산에서 가장 중요한 순간?

**초과 인지** > 진행률 확인 > 예산 설정 순서.
- 예산 설정은 월 1회. 중요하지만 자주 발생하지 않음.
- 진행률 확인은 가끔. "현재 70% 썼네" 정도.
- **초과 인지가 핵심**: "식비 예산 초과!" → 프로그레스 바가 빨간색으로 변하는 순간이 **행동 변화의 트리거**. 이 순간을 놓치면 예산 기능의 가치가 0이 된다.

### Q30. 카테고리별 + 전체 예산을 함께 지원하는 이유?

- **전체 예산**: "이번 달 총 100만원 이내로 쓰자" → 큰 그림
- **카테고리별**: "식비 30만, 교통 5만, 쇼핑 10만" → 세부 통제
- **함께 필요한 이유**: 전체만 있으면 "어디를 줄여야 하는지" 모름. 카테고리만 있으면 "전체적으로 괜찮은지" 모름. 둘 다 있어야 **전략적 소비 관리** 가능.

### Q31. 자산과 부채를 같은 축에서 관리하는 이유?

**순자산 = 자산 − 부채**. 이 공식을 직관적으로 보여주려면 같은 화면에 있어야 한다.
- 자산(은행 잔고 500만) + 부채(카드대금 -80만) = **순자산 420만원**
- 부채를 별도 메뉴로 숨기면 "내 재정 상태"를 한눈에 파악할 수 없음
- `NetWorthCard` 컴포넌트가 자산/부채를 합산한 **순자산 카드**를 최상단에 표시

### Q32. 거래-계좌 연결이 만드는 추가 가치?

**"어디서 나간 돈인지" 추적 가능.**
- 단순 가계부: "식비 9000원" → 끝
- 계좌 연결: "식비 9000원, 카카오뱅크에서 출금" → 계좌 잔액 자동 업데이트
- **가치**: 은행 앱을 열지 않아도 가계부 안에서 잔액을 확인 가능. "카드 대금은 얼마지?", "통장에 얼마 남았지?"를 앱 내에서 해결.

### Q33. 자동 잔액 변동의 신뢰와 부담?

| 측면 | 내용 |
|------|------|
| **신뢰** | 거래를 기록하면 잔액이 알아서 바뀜 → "앱이 내 재정을 정확히 반영하고 있다"는 신뢰감 |
| **부담** | 잔액이 실제 은행 잔고와 다를 수 있음 → "이 숫자가 맞나?" 의구심 |
| **해결** | 잔액 직접 수정 기능 제공 (계좌 편집 Drawer) + 향후 은행 API 연동으로 자동 동기화 (Phase 18B) |

## Part F. 운영과 우선순위 (6문)

### Q34. 금융 데이터를 AI에 보내는 서비스에서 사용자가 가장 불안해할 지점?

> **"내 결제 내역이 AI 회사 서버에 저장되지 않을까?"**

대응:
- `originalInput` AES-256-GCM 암호화 저장 → DB 유출 시에도 원문 노출 불가
- LLM API 호출은 **stateless** → Kimi/Fireworks는 대화를 저장하지 않음 (API TOS)
- **향후 개선**: 개인정보처리방침에 "AI 파싱은 실시간 처리 후 삭제, 학습에 사용하지 않음" 명시

### Q35. 이메일 로그인만 남긴 영향?

- **부정적**: Google/카카오 소셜 로그인 대비 가입 **전환율 20~30% 하락** 예상 (비밀번호 입력 + 이메일 인증 번거로움)
- **긍정적**: 커스텀 도메인 없이 제3자 OAuth를 구성하기 어려움 → 생략함으로써 운영 복잡도 감소
- **현실적 판단**: 개인 사이드 프로젝트에서 OAuth 인증 서버(Google/카카오 앱 등록, redirect URI 관리)를 유지하는 비용 대비, 이메일 로그인의 단순함이 우세

### Q36. 모바일 퍼스트 + 네이티브 래핑을 택한 이유?

- **사용 맥락**: 가계부는 결제 직후(모바일)에 기록하는 앱. 데스크톱에서 쓸 일이 거의 없음.
- **개발 효율**: 웹 기술(React/Next.js) 하나로 웹+iOS+Android 커버 → 네이티브 앱 별도 개발 비용 제거
- **Capacitor WebView 방식**: Vercel URL을 WebView로 로드 → 웹 배포 시 앱도 자동 업데이트 (앱스토어 재배포 불필요)
- **트레이드오프**: 오프라인 사용 불가, 네이티브 성능 부족 → 현재 사용 시나리오에서는 허용 가능

### Q37. 미완료 기능 중 무엇을 먼저 출시?

| 순위 | 기능 | 이유 |
|------|------|------|
| 🥇 | **음성 입력** | 기존 입력 파이프라인에 Web Speech API만 연결. 개발 비용 낮음 + "말로 가계부 쓰기" 마케팅 가치 높음 |

### Q38. 3개월 더 운영 시 가장 먼저 검증하고 싶은 가설?

> **"자연어 AI 파싱이 수동 입력 대비 D7 리텐션을 2배 이상 높인다."**

검증 방법:
- A/B 테스트: 그룹 A(AI 파싱만 제공) vs 그룹 B(수동 입력만 제공)
- 측정: D1/D3/D7 리텐션, 월 평균 거래 수, 입력 소요 시간
- **이 가설이 틀리면**: AI 파싱은 "신기한 데모"일 뿐 실제 습관화에 기여하지 않는 것 → 제품 방향 재검토 필요
